# 常见问题 FAQ

## 关于品牌和数据

### Q1: 品牌和店铺有什么区别？

**A:** 这是本项目最核心的概念区分。

- **品牌**：眼镜的制造商品牌，如GUCCI、RIGARDS、999.9、万年龟、增永等
- **店铺**：销售渠道，如午山匠造、上野眼镜、目下、镜架收藏社等

**特殊情况**：有些名称既是店铺也是品牌，如"万年龟"、"增永"。需要根据商品内容判断：
- 如果商品名称是"万年龟 M2048 日本手工眼镜"，Brand字段应该是"万年龟"（制造商品牌）
- 如果商品名称是"万年龟店铺 GUCCI GG1829SK"，Brand字段应该是"GUCCI"（制造商品牌）

### Q2: 为什么商品价格全为0？

**A:** 这是淘宝"店铺透视"功能导出数据的限制，不是数据缺失。

淘宝不会在导出数据中包含真实价格，需要通过其他方式补充（如爬虫、API、手动录入）。

### Q3: 识别率多少才算合格？

**A:** 根据甲方Persona（20年买手经验）的标准：
- **品牌识别率**：95%+（目前57.9%）
- **型号识别率**：80%+（目前23.9%）

这是业务需求，不是技术指标。识别率低会影响采购决策和品牌管理。

### Q4: 为什么不能达到100%识别率？

**A:** 原因有三：
1. 商品名称格式不统一（有的品牌+型号紧贴，有的用空格分隔）
2. 有些商品没有标准的型号代码
3. 有些品牌不在品牌字典中（新品牌、小众品牌）

**解决方案**：自动识别 + 人工补充。先用算法识别大部分，剩下的由买手手动补充。

## 关于数据清洗

### Q5: 什么是"非商品项"？

**A:** 不是真实商品的数据，如：
- 配镜流程
- 维修服务
- 店铺公告
- 优惠券

这些数据需要过滤掉，不应该出现在商品库中。

### Q6: 如何判断一个商品是否是"非商品项"？

**A:** 基于关键词匹配：
- 包含"配镜"、"维修"、"服务"、"流程"等关键词
- 付款人数为0且商品名称异常
- 没有品牌和型号信息

### Q7: 数据清洗的标准流程是什么？

**A:** 
1. 数据预处理（去重、格式化）
2. 品牌识别（基于品牌字典和正则表达式）
3. 型号提取（基于品牌特定规则）
4. 过滤非商品项
5. 质量检查（计算识别率、缺失率）
6. 多Persona评审
7. 根据反馈优化算法
8. 重新清洗，直到达到质量标准

## 关于Persona系统

### Q8: 为什么要用Persona评审？

**A:** 单一视角容易遗漏问题。

- 小M（我）只关注技术实现，可能忽略业务价值
- 甲方关注品牌准确性，可能忽略数据完整性
- 电商分析师关注销售数据，可能忽略品牌分类

通过多Persona评审，可以从不同角度发现问题，提升数据质量。

### Q9: 7个Persona分别负责什么？

**A:**
- **小M**：执行数据清洗，协调多Persona评审
- **甲方**：定义质量标准，评审品牌准确性
- **专家1（奢侈品采购顾问）**：评审品牌字典和品牌识别
- **专家2（电商数据分析师）**：评审数据完整性和商业价值
- **专家3（眼镜行业品牌经理）**：评审品牌分类和品牌管理
- **专家4（零售运营经理）**：评审运营指标和决策支持
- **专家5（数据治理专家）**：评审数据标准化和治理体系

### Q10: Persona是真人还是AI？

**A:** Persona是AI模拟的专业角色。

每个Persona有自己的专业背景、评审标准、关注点。通过模拟不同专业角色，可以从多角度评审数据质量。

## 关于Node架构

### Q11: 什么是Node？

**A:** Node是一个独立的处理单元，有统一的结构：
- **meta**：Node的基本信息（id、name、role、type）
- **input**：Node接收什么数据
- **process**：Node如何处理数据
- **output**：Node输出什么数据
- **feedback**：Node如何与其他Node协作

### Q12: 为什么要用Node架构？

**A:** 基于奥卡姆剃刀原则，用最简单的结构描述复杂的系统。

- 统一的结构，易于理解和维护
- 灵活的组合，适应不同场景
- 清晰的连接，形成协作网络

### Q13: Node架构可以应用于其他场景吗？

**A:** 可以。Node架构是通用的，不仅适用于数据清洗，还可以应用于：
- 内容创作（写作者 + 编辑者 + 审核者）
- 产品设计（设计者 + 评审者 + 用户测试者）
- 决策支持（分析者 + 专家 + 决策者）
- 任何需要多角色协作的场景

## 关于协作方式

### Q14: 用户和小M的协作模式是什么？

**A:**
1. **用户定方向**：告诉我要做什么、为什么做
2. **我做执行**：设计方案、写代码、跑数据
3. **用户验收**：检查结果是否符合预期
4. **我迭代优化**：根据反馈改进

### Q15: 如何降低沟通成本？

**A:** 通过固化关键信息到GitHub：
- 对话风格指南：避免重复说明沟通偏好
- 常见问题FAQ：避免重复回答相同问题
- 决策记录：记录重要决策和原因
- 工作流程：标准化协作流程

### Q16: 遇到问题怎么办？

**A:**
1. 先尝试自己解决（查文档、看代码）
2. 如果解决不了，及时向用户反馈
3. 说明问题、原因、尝试过的方案
4. 提出建议，等待用户决策
